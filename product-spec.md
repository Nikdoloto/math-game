# Спецификация MVP: платформа обучающих игр и игра "Адаптивный гараж"

Источник правды:

- `product-vision.md`
- `math-games deep-research-report.md`

## 1. Вижн и цель

Мы создаем простую веб-платформу обучающих игр для детей. Первая игра внутри платформы - "Адаптивный гараж", тренировка таблицы умножения для детей 7-10 лет.

Продукт должен быть доступен по ссылке, удобно работать на телефоне и размещаться на собственном сервере через Nginx. На MVP не используем базу данных: прогресс игрока хранится в браузере. Это упрощает запуск и поддержку, но ограничивает настоящие социальные функции: общий лидерборд между разными устройствами потребует отдельного серверного слоя позже.

Главная продуктовая цель MVP: ребенок за 7-12 минут проходит приятную тренировку, видит понятный прогресс по таблице умножения и хочет вернуться завтра.

## 2. Пользователи

Основной пользователь: ребенок 7-10 лет.

Контекст:

- играет с телефона, планшета или компьютера;
- может открыть игру дома, на перемене или после школы;
- может поделиться ссылкой с одноклассниками;
- хочет видеть себя как игрока: никнейм, прогресс, достижения.

Взрослый пользователь не является главным фокусом MVP. Раздел "Мой прогресс" должен быть понятен ребенку и при этом достаточно ясен взрослому, если он посмотрит экран вместе с ребенком.

## 3. Продуктовые границы MVP

### Входит в MVP

- Статическая веб-платформа, размещаемая через Nginx.
- Главный экран платформы со списком игр.
- Одна доступная игра: "Адаптивный гараж".
- Ввод никнейма при первом входе.
- Сохранение никнейма и прогресса в браузере.
- Мобильная адаптация интерфейса.
- Короткие игровые сессии на 7-12 минут.
- 2-4 раунда по 60-90 секунд.
- Адаптивный подбор примеров по конкретным фактам умножения.
- Очередь ошибок с повторным предъявлением.
- Нейтральная мгновенная обратная связь.
- Карта прогресса по фактам или семействам фактов.
- Экран "Мой прогресс".
- Приятная детская визуальная оболочка с легким Minecraft-настроением: блочный мир, объемные элементы, гараж, транспорт, кубические детали.
- Шаринг: возможность скопировать ссылку на игру и/или текстовый результат с никнеймом и достижением.
- Локальный "дружеский рейтинг" на устройстве для демонстрации механики.

### Не входит в MVP

- База данных.
- Авторизация по email, телефону или соцсетям.
- Настоящий общий лидерборд между разными устройствами.
- Live PvP.
- Чат между детьми.
- Платежи, подписки, энергия, сезоны, боевые пропуски.
- Сложный родительский или учительский кабинет.
- Большой открытый мир.

## 4. Важное техническое решение по прогрессу и лидерборду

### Решение для MVP

Прогресс хранится в `localStorage` браузера. Это значит:

- ребенок возвращается на том же устройстве и видит свой прогресс;
- проект можно разместить как статический сайт на Nginx;
- не нужно поднимать backend, базу данных и админку;
- данные не синхронизируются между устройствами;
- общий лидерборд между одноклассниками не может быть настоящим без серверного хранения.

### Как закрываем социальный сценарий в MVP

В MVP делаем легкий социальный слой без базы:

- кнопка "Поделиться игрой" копирует ссылку;
- экран результата предлагает текст: "Я закрыл 8 новых примеров в Адаптивном гараже. Попробуй обогнать мой прогресс";
- локальный рейтинг показывает результаты профилей, созданных в этом браузере, и может использоваться как прототип будущего лидерборда.

### Следующий слой после MVP

Если нужен настоящий лидерборд между одноклассниками, понадобится минимальный backend:

- простое API для отправки результата;
- хранение никнейма, score и даты;
- антиспам-ограничения;
- модерация или фильтр никнеймов;
- лучше начать с SQLite или простого JSON-хранилища на сервере, но это уже не "статический Nginx only".

## 5. Пользовательские сценарии

### Сценарий 1. Первый вход

1. Ребенок открывает ссылку.
2. Видит главный экран платформы.
3. Вводит никнейм.
4. Переходит к списку игр.
5. Видит игру "Адаптивный гараж" и нажимает "Играть".

Критерии приемки:

- никнейм сохраняется после перезагрузки страницы;
- никнейм можно изменить в настройках;
- без никнейма игра не блокируется навсегда: можно быстро ввести новый;
- на мобильном экране ввод и кнопки удобно нажимаются пальцем.

### Сценарий 2. Игровая сессия

1. Ребенок нажимает "Поехали".
2. Игра запускает короткий раунд.
3. На экране появляется пример, например `6 x 7`.
4. Ребенок вводит ответ.
5. Игра сразу показывает реакцию.
6. Если ответ неверный, факт попадает в очередь повторения.
7. После нескольких раундов ребенок видит экран итогов.

Критерии приемки:

- сессия имеет понятный старт и финиш;
- неверный ответ не выглядит как наказание;
- ошибочный факт появляется повторно позже в той же или следующей сессии;
- ребенок видит, что именно улучшилось.

### Сценарий 3. Мой прогресс

1. Ребенок открывает "Мой прогресс".
2. Видит карту фактов умножения.
3. Понимает, какие факты уже уверенные, какие требуют практики.
4. Видит короткие достижения.

Критерии приемки:

- прогресс отображается по конкретным фактам, а не только общим уровнем;
- есть понятные статусы: "тренируем", "почти готово", "уверенно";
- экран читается на телефоне без горизонтального скролла;
- прогресс сохраняется после закрытия браузера.

### Сценарий 4. Поделиться с одноклассником

1. Ребенок открывает экран итогов или главный экран.
2. Нажимает "Поделиться".
3. Получает ссылку или текст результата.
4. Отправляет другу вне приложения.

Критерии приемки:

- кнопка работает на desktop и mobile;
- если Web Share API недоступен, есть fallback: копирование текста в буфер;
- ссылка открывает платформу;
- текст не содержит персональных данных, кроме выбранного никнейма.

## 6. Игровой цикл

Базовый цикл "Адаптивного гаража":

1. Платформа открыта.
2. Игрок выбирает игру.
3. Машинка/транспорт стоит в гараже.
4. Игрок запускает короткий заезд.
5. Каждый правильный ответ дает движение, энергию или деталь для транспорта.
6. Ошибка мягко подсвечивает правильную связку и отправляет пример в повтор.
7. После раунда гараж обновляется: появляется новая деталь, улучшение или визуальный след прогресса.
8. Игрок видит "Мой прогресс" и может поделиться результатом.

Важно: игровая оболочка не должна съедать учебный цикл. Между примерами должно быть минимум лишних переходов.

## 7. Обучающий алгоритм MVP

### Факт

Факт - конкретная пара множителей, например:

- `6 x 7`
- `8 x 4`
- `9 x 9`

Для MVP можно считать `6 x 7` и `7 x 6` связанными фактами, но хранить статистику по каждому отображаемому варианту отдельно или через общий canonical key `6x7`.

### Статусы факта

- `new`: еще почти не тренировался.
- `learning`: были попытки, точность или скорость нестабильны.
- `review`: в целом получается, но нужен повтор.
- `mastered`: высокая точность и приемлемое время ответа.

### Данные по факту

Для каждого факта храним:

- количество попыток;
- количество правильных ответов;
- количество ошибок;
- среднее время ответа;
- дату/время последней попытки;
- текущий статус;
- счетчик успешных повторов подряд.

### Подбор примеров

Приоритет выдачи:

1. Факты из очереди ошибок.
2. Факты со статусом `learning`.
3. Факты, которые давно не повторялись.
4. Новые факты.
5. Иногда mastered-факты для поддержания уверенности.

### Обработка ошибки

Если ребенок ошибся:

1. Показать правильный ответ коротко и спокойно.
2. Добавить факт в error queue.
3. Повторить факт через 2-5 других примеров.
4. Не снижать общий результат агрессивно.
5. В итогах показать как "факт для тренировки", а не как провал.

## 8. Визуальный стиль

Цель: игра должна ощущаться как детская объемная история, а не как обычный учебный интерфейс.

Направление:

- легкий Minecraft-vibe: кубические формы, блоки, гараж, детали, дорога, ресурсы;
- 2.5D или простая 3D-подача: объемные карточки, изометрия, тени, слои, параллакс;
- приятные материалы: дерево, металл, резина, цветные кубики;
- ярко, но не кислотно;
- интерфейс дружелюбный, крупный, с хорошими отступами;
- на mobile все основные действия доступны большим пальцем.

Рекомендация по реализации:

- Для MVP не делать полноценную сложную 3D-игру.
- Использовать CSS 3D, изометрические иллюстрации, легкие canvas/WebGL-эффекты или готовые простые 3D-ассеты.
- Если используется Three.js, сцена должна быть небольшой: гараж, машинка, несколько блоков, простые анимации.
- Учебный слой должен оставаться быстрым и читабельным.

Обязательные визуальные элементы:

- главный экран платформы с карточкой игры;
- гараж или мастерская;
- транспорт/машинка как главный объект прогресса;
- карта фактов в виде понятной сетки;
- экран результата с приятной анимацией улучшения;
- кнопка поделиться.

## 9. UI-структура

### Экраны платформы

- Welcome / ввод никнейма.
- Главный экран платформы со списком игр.
- Настройки игрока: никнейм, сброс локального прогресса.

### Экраны игры

- Экран гаража.
- Экран выбора режима: обычная тренировка, спокойный режим без таймера.
- Игровой раунд.
- Итоги сессии.
- Мой прогресс.
- Локальный рейтинг / достижения.

### Mobile-first требования

- Минимальная целевая ширина: 360 px.
- Все основные кнопки не меньше 44 px по высоте.
- Цифровой ввод должен быть удобен на телефоне.
- Нельзя полагаться только на hover-состояния.
- Важные элементы не должны быть у нижнего края, где их перекрывают браузерные панели.
- Текст внутри кнопок не должен обрезаться.

## 10. Архитектура

### Рекомендуемый стек

Для MVP лучше выбрать простой статический frontend:

- Vite как сборщик;
- React или чистый TypeScript/JavaScript;
- CSS Modules или обычный CSS;
- localStorage для прогресса;
- Nginx для раздачи собранной папки `dist`.

Пояснение: Vite - инструмент, который собирает frontend в обычные статические файлы. Nginx сможет раздавать эти файлы без Node.js на сервере.

Если хочется еще проще, можно сделать без React, но React удобнее для экранов, состояний и будущего расширения платформы.

### Без backend в MVP

В MVP нет API и базы данных.

Сайт работает так:

- браузер загружает HTML/CSS/JS;
- приложение читает прогресс из localStorage;
- приложение обновляет localStorage после сессии;
- Nginx только отдает статические файлы.

### Развертывание на Nginx

Ожидаемый результат сборки:

- папка `dist`;
- внутри статические файлы `index.html`, JS, CSS, ассеты;
- Nginx настроен на отдачу `index.html` для всех маршрутов приложения.

Пример ожидаемой Nginx-логики:

```nginx
location / {
  try_files $uri $uri/ /index.html;
}
```

## 12. Критерии успеха MVP

MVP можно считать успешным, если:

- ребенок может пройти полную сессию на телефоне;
- прогресс сохраняется после перезагрузки страницы;
- игра подбирает не только случайные примеры, а учитывает ошибки и слабые факты;
- после ошибки факт возвращается позже;
- экран "Мой прогресс" показывает понятную карту фактов;
- визуально продукт ощущается как детская игра с объемной средой, а не как форма с примерами;
- проект можно собрать в статические файлы и разместить на Nginx;
- есть честное понимание, что настоящий общий лидерборд требует следующего технического слоя.

## 14. Тестирование

### Что проверить вручную

- Первый вход и сохранение никнейма.
- Запуск сессии.
- Правильный ответ.
- Неправильный ответ.
- Возврат ошибочного факта.
- Завершение сессии.
- Сохранение прогресса после перезагрузки.
- Сброс прогресса.
- Работа на ширине 360 px.
- Работа на desktop.
- Кнопка "Поделиться".
- Сборка проекта.

### Автоматические тесты, если добавляем

Минимум:

- функция выбора следующего факта;
- функция обновления статистики после ответа;
- функция расчета статуса факта;
- сериализация и восстановление localStorage-данных.

## 16. Стиль кода

- Логика обучения отдельно от визуальных компонентов.
- Scheduler и progress engine должны быть чистыми функциями, чтобы их было легко тестировать.
- Компоненты UI не должны напрямую знать детали localStorage.
- Все ключи localStorage хранить в одном месте.
- Не смешивать платформенный код и код конкретной игры без необходимости.
- Названия переменных на английском, пользовательские тексты на русском.

Пример:

```ts
type FactStatus = "new" | "learning" | "review" | "mastered";

type FactProgress = {
  attempts: number;
  correct: number;
  errors: number;
  avgMs: number;
  streak: number;
  status: FactStatus;
  lastSeenAt?: string;
};
```

## 17. Границы

### Можно делать без дополнительного согласования

- Статический frontend.
- localStorage для прогресса.
- Простые визуальные эффекты и анимации.
- Mobile-first верстку.
- Локальный рейтинг как прототип.
- Улучшения формулировок детского интерфейса.

### Нужно спросить перед реализацией

- Добавление backend.
- Добавление базы данных.
- Реальная авторизация.
- Сбор персональных данных ребенка.
- Публичный глобальный лидерборд.
- Чат, комментарии или любые детские социальные коммуникации внутри продукта.
- Сложная 3D-сцена, которая может сильно увеличить время разработки.

### Нельзя делать в MVP

- Хранить персональные данные без необходимости.
- Делать публичный рейтинг с реальными именами детей.
- Наказывать ребенка за ошибки визуально или механически.
- Ставить скорость выше точности.
- Прятать учебный цикл за длинной игровой метой.

## 18. Риски

| Риск | Почему важно | Как снижаем |
|---|---|---|
| Нет настоящего общего лидерборда без backend | Пользователь хочет делиться и соревноваться | В MVP делаем шаринг и локальный рейтинг, backend выносим в следующий этап |
| Игра станет просто учебной формой | Ребенку будет скучно | Делаем объемный гараж, транспорт, анимации, награды |
| 3D станет слишком дорогим | MVP может затянуться | Используем 2.5D/CSS/легкий Three.js, без сложного мира |
| Таймер усилит тревожность | Часть детей будет избегать игры | Таймер скрыт по умолчанию |
| localStorage можно потерять | Прогресс привязан к браузеру | Честно фиксируем ограничение, позже можно добавить серверный профиль |
